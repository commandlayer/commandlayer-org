Runtime

Execution on top of stable schemas — guarantees, policy, and economics.

Commons and Commercial define what a verb means (request + receipt shapes). The Runtime defines how execution happens: routing, verification, pricing, rate limits, retries, idempotency, audit logs, and end-to-end guarantees — while keeping the semantic contract stable.

What Runtime is (and isn’t)

Runtime is not “the schemas.” It’s the part you pay for when you want outcomes, guarantees, and production-grade execution.

  • Is: execution endpoints + policy + proofs + economics.
  • Is: stable receipts, auditability, and reliability controls.
  • Is not: a competing verb language (Commons/Commercial already do that).
  • Is not: a closed semantic system — contracts remain public and portable.

Where Runtime sits in the stack

Runtime is the “execution plane.” It consumes the same pinned schemas used everywhere else, and returns the same receipts — but adds operational guarantees so agents can be used like infrastructure.

Inputs

A verb invocation (Commons or Commercial) shaped exactly by the public JSON Schemas. Routing is discovered via Agent Cards and ENS metadata.

Runtime policy

Verification, authn/authz, idempotency keys, rate limits, retries/timeouts, safety filters, and vendor/model selection — without changing the request shape.

Outputs

Typed receipts with trace IDs and standardized status/usage. Same receipt contracts across agents, runtimes, and payment rails.

What you get from Runtime

This is the “why pay” layer: predictable outcomes and guarantees you can build businesses on, while the verb language remains public goods.

Guarantees & controls

Runtime adds enforceable operational behavior around schema-stable calls.

  • Idempotency: safe retries for commercial flows (authorize/checkout/purchase/ship/verify).
  • Traceability: consistent trace_id threading across steps and services.
  • Policy: safety, compliance, allowlists/denylists, and “who can call what” rules.
  • Reliability: timeouts, circuit breakers, retry budgets, and structured errors.
  • Audit: receipts + logs that can be checked later without reinterpreting payloads.

Economics: value lives in execution

Commons/Commercial schemas are always accessible. Runtime is where pricing, metering, and settlement live.

  • Metering: standardized usage fields in receipts enable predictable billing.
  • Pricing tiers: pay for SLA, throughput, latency targets, and policy bundles.
  • Settlement: designed to sit cleanly on payment rails (x402-style envelopes later).
  • Portability: if you self-host execution, you still keep the same verb contracts.

The forward-looking bet: the world standardizes on a shared verb language. The winners are the runtimes that can execute it reliably, prove what happened, and settle economically — without fragmenting semantics.

Runtime-style receipt envelope

Runtime returns the same typed receipts — plus operational fields you can rely on for billing, audit, and debugging.

Example receipt (illustrative)

This is the kind of stable “shape” you can build tooling and businesses around. 

{
  "x402": {
    "verb": "checkout",
    "version": "1.0.0"
  },
  "trace": {
    "trace_id": "c0d48ff3-e20a-4be4-922f-7bf0478da3cb",
    "started_at": "2025-12-12T20:19:00.000Z"
  },
  "status": {
    "code": "ok",
    "message": "checkout complete"
  },
  "result": {
    "checkout_id": "chk_1734034740",
    "total_amount": 129.99,
    "currency": "USD"
  },
  "usage": {
    "units": 1,
    "compute_ms": 642,
    "billable": true
  }
}

The key point: clients don’t guess what happened. They read a typed receipt, tied to a known schema version and a trace ID.

Runtime roadmap (high level)

Keep this page honest: Runtime is “coming online.” Ship iteratively without overpromising.

  • Phase 1: reference endpoints + receipts + trace IDs (demo-grade → prod-hardening).
  • Phase 2: policy bundles (rate limits, auth, allowlists) + metering in receipts.
  • Phase 3: multi-runtime portability: same contracts, different executors.
  • Phase 4: settlement + proofs on standardized rails (as ecosystem norms mature).

If you want a simple mental model: Commons/Commercial are the public language. Runtime is the business. You can swap executors without breaking the contract.
