Receipts

A CommandLayer receipt is not a log entry. It is a cryptographic artifact that proves what an agent, runtime, or external system did — and lets anyone verify it independently after the fact.

What a receipt proves

Receipts provide tamper-evident evidence of normalized actions/events, signer identity, and verification outcome.

Required proof fields

CommandLayer verification depends on canonicalization (json.sorted_keys.v1), hashing (SHA-256), and signature validation (Ed25519).

metadata.trace

Trace metadata links execution context across runtime, services, and integration boundaries for auditability.

metadata.proof

Includes signer and signature material used by verifiers, including signer identity like runtime.commandlayer.eth, public_key_source: ens_txt, key_id: vC4WbcNoq2znSCiQ, and verification signals such as ens_resolved: true, hash_matches: true, and signature_valid: true.

Verification lifecycle

Resolve signer key → canonicalize payload → recompute hash → verify signature → return deterministic validity.

Trust Verification verbs

verify, authenticate, authorize, attest, sign, permit, grant, approve, reject, endorse.

Integration receipt types

Receipts can describe external events (for example webhook observations) and paid actions while preserving canonical proof semantics.

VerifyAgent

VerifyAgent is the reference verifier for CommandLayer receipts. It does not sign receipts, execute actions, or hold runtime keys. It verifies proof produced by signers such as runtime.commandlayer.eth.

Verification warning

Schema validation checks shape. Proof verification checks truth. A schema-valid receipt can still fail verification if the hash or signature is wrong.
