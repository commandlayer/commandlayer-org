How CommandLayer Verification Works

Agents create signed receipts. VerifyAgent checks the receipt, signer, hash, signature, and trust semantics.

Five-step verification flow

  1. Agent executes an action.
  2. SDK emits a signed receipt.
  3. Receipt carries proof metadata.
  4. VerifyAgent runs Parse → Schema → Hash → Signature → Signer.
  5. Systems accept or reject the action based on verification checks.

Layer-by-layer explanation

  • CLAS: action and receipt semantics.
  • SDK: creates receipts.
  • VerifyAgent: independently verifies receipts.
  • ENS/ERC-8004: optional discovery and identity.
  • MCP/OpenAPI: invocation surfaces.

Manual Verification

Paste a receipt into the verifier to inspect each check and verdict.

Ambient Verification

Use verification URLs, embedded badges, and webhooks so verification runs automatically where actions are consumed.

Important verification semantics

Schema-valid does not always mean cryptographically valid.
ENS/ERC-8004 are discovery and identity layers, not hard dependencies.
VerifyAgent is a reference verifier, not the only possible verifier.
