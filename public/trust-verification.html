CLAS Trust Verification v1

A machine-readable standard for verifiable actions, approvals, signatures, permissions, attestations, and trust decisions.

Canonical verbs

verify

Confirm whether proof data and trust semantics are valid for an action.

authenticate

Establish who produced the action or proof artifact.

authorize

Determine whether an actor is allowed to perform the requested action.

attest

Issue a claim that a condition, fact, or action outcome is true.

sign

Apply a cryptographic signature over canonicalized action evidence.

permit

Represent policy-based permission to continue an action flow.

grant

Assign explicit rights, capabilities, or scopes to another party.

approve

Accept a request after checks, policy, or workflow requirements pass.

reject

Deny a request when checks, policy, or constraints are not satisfied.

endorse

Publicly support a claim or signer based on trust and verification evidence.

Shared proof model

  • canonicalization: deterministic normalization that guarantees stable bytes for hashing and signing.
  • hash: digest of canonicalized action payload used as the integrity anchor.
  • signature_alg: declared signing algorithm used to verify the proof.
  • signature: cryptographic output proving possession of the signer key.
  • key_id: key reference used to resolve and validate the expected public key.
  • signer: identity string representing the actor that issued the signed proof.

Verification pipeline

Parse → Schema → Hash → Signature → Signer

Schema validation is necessary for structural correctness, but cryptographic verification is required for trust-grade validity.

Relationship to VerifyAgent

VerifyAgent is the reference verifier for CLAS Trust Verification receipts and demonstrates one portable implementation of verification semantics.

VerifyAgent is a reference verifier, not the only possible verifier.

Relationship to ENS / ERC-8004

  • CLAS is network-agnostic and can operate across different identity and execution environments.
  • ENS/ERC-8004 can be used for identity, discovery, reputation, and validation records.
  • ENS/ERC-8004 are not hard dependencies for CLAS trust semantics.
