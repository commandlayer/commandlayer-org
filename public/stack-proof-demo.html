Canonical Stack Proof Demo (local/dev)

This page documents the developer-facing proof flow for CommandLayer in local/dev environments. It is not a claim that production runtime endpoints are currently reachable.

Canonical flow

MCP sign request → runtime signs receipt → VerifyAgent verifies → tampered receipt invalidates

Canonical proof envelope

{
  "receipt": {
    "request_id": "req_demo_123",
    "agent": "runtime.commandlayer.eth",
    "verb": "summarize",
    "result": {
      "summary": "CommandLayer signs verifiable runtime receipts."
    }
  },
  "metadata": {
    "proof": {
      "canonicalization": "json.sorted_keys.v1",
      "hash": {
        "alg": "SHA-256",
        "value": "1e26...demo...f9"
      },
      "signature": {
        "alg": "Ed25519",
        "value": "z6Mkh...demo-signature"
      }
    }
  }
}

Expected verification outcomes

  • VerifyAgent result: VERIFIED
  • Tampered result: INVALID

Run it locally

  • runtime: run npm test; use your local sign endpoint to emit signed receipts in dev.
  • verifyagent: run npm run e2e:runtime against locally generated receipts.
  • mcp-server: run npm test; use the local sign bridge for MCP→runtime handoff.

Architecture (text diagram)

CLAS            = contract
runtime-core    = crypto truth
runtime         = signer / executor
MCP             = bridge
VerifyAgent     = verifier
agent-sdk       = developer wrapper

Warnings

  • MCP is not the trust root.
  • schema-valid is not verified. Verification requires canonical hash + signature checks.
  • Production runtime must use a key matching ENS TXT records.

Production availability should not be claimed as live until runtime.commandlayer.org is publicly reachable.
