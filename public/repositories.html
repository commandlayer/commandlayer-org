Repositories

CommandLayer is one system with separate repos for separate layers. Read this page top-down: Commons and Commercial define the contract, Agent Cards define discovery, runtimes prove execution, and public surfaces explain the stack.

Protocol stack

Read the repositories in stack order: contract first, discovery second, execution third, then SDK and publishing surfaces that mirror and explain that core rather than competing with it.

contract

protocol-commons

The canonical open contract layer. This repo defines the public request and receipt schemas, the minimum verifiable receipt, and the baseline semantics builders should target first.

protocol-commercial

The commerce-oriented contract extension. It adds payment, checkout, fulfillment, shipping, and verification schemas while preserving the Commons-first model and portable receipt discipline.

agent-cards

The discovery and identity layer. Agent Cards say who an agent is, which verbs and versions it supports, and where clients should route requests without redefining the contract itself.

Architecture rule

Do not read these repos as separate products. They are one stack with a strict boundary: protocol repos define meaning, discovery binds identity, execution proves what happened, SDKs package access, and the website mirrors/publishes the approved public story.

stack rule

Canonical receipt first

Commons and Commercial define the canonical request/receipt surfaces. Agent Cards describe discovery and routing. Runtime layers may add execution evidence, but proof fields sit around canonical contracts rather than inside the contract definition layer.
